## Vendor Status
1. Active / *mangkal* = can be followed AND can be called
2. Moving / *jalan* = can be followed BUT cannot be called
3. Idle / *istirahat* = can be followed BUT cannot be called
4. Close / *tutup* = cannot be followed AND cannot be called

## Clustering Lifecycle
1. Forming / *request*  : <br /> - Pre-early stages, not valid yet, called as an **request** or *craving location* pointed with lat and long. <br /> - To make it active / assigned cluster, there must be minimum 2 requests at a certain of time and within 10 meters at maximum.
2. Assigned / Active Cluster : <br /> - Server automatically assign an active cluster to the selected vendor. <br /> - But the vendor has to accept or reject it first.
4. Accepted : <br /> Vendor already accept the cluster and now can follow the cluster **centroid**.
5. Rejected :  <br /> Vendor rejected the cluster. 
6. Fulfilled : <br /> Vendor arrived / done.
7. Expired : <br /> Dead because of time.
9. Dissolved : <br /> Forced to be offed because of unexpected things.

- Unfortunately, the customer can only call max. 3 vendors to prevent vendor call spamming.
- This constraint will be reseted if a **Request** is **expired** or the **Cluster** is **rejected** or **fulfilled** or **expired** or **dissolved** 


###  2. Follow / *ikuti* Vendor 
1. Customer can see the static locatiom of the vendors that exist nearby them.
2. If vendor status is ACTIVE or MOVING or IDLE, then the Customer can follow the selected vendor.
3. Customer and vendor can see each other real-time location and status

NB: For now, the position of nearby vendors on the map are static and updated only based on interval from Frontend via HTTP GET Request, we still have no clue how to make it technically possible for efficient real-time operations.

## Where and How is the DBSCAN Clustering process is happening? 

The DBSCAN Clustering itself is happening when:
1. Customer decides to call a vendor
2. There are more than 2 customers within 10m radius 

How it works?
1. It works like k-means clustering,
2. BUT k-means clustering works with choosing: how manyclusters to exist, instead, DBSCAN works choosing the minimum of hotspots and maximum distance between hotspots.
3. Take this as an example: Epsilon = 0.001 and requests = 2. it means there must be 2 requests within 10 meters range.
